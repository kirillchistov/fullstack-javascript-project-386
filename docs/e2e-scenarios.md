# Интеграционные (e2e) сценарии

Проверяются в реальном браузере (Playwright + Chromium) против полного стека:
Fastify-бэкенд с хранилищем в памяти + Vite-фронтенд. Оба сервера поднимаются
автоматически при запуске тестов (`webServer` в [e2e/playwright.config.ts](../e2e/playwright.config.ts)).

Запуск: `cd e2e && npm test` (один раз перед этим — `npx playwright install chromium`).

## Зафиксированные сценарии

Реализация: [e2e/tests/booking.spec.ts](../e2e/tests/booking.spec.ts).

| № | Сценарий | Шаги | Проверяемый результат |
|---|---|---|---|
| 1 | Основной путь бронирования (гость) | Открыть страницу → увидеть типы событий → выбрать дату в календаре → выбрать свободный слот → заполнить имя, email, комментарий → подтвердить | Экран «Встреча забронирована» с датой, временем слота и email гостя |
| 2 | Занятый слот исчезает | Забронировать слот → вернуться к выбору времени на ту же дату | Забронированного времени нет в списке свободных слотов |
| 3 | Конфликт: слот уже занят | Два гостя видят один слот → первый бронирует → второй бронирует тот же слот из устаревшего списка | Второй гость видит понятную ошибку «Этот слот уже занят» (409 из контракта) и обновлённый список без этого слота |
| 4 | Владелец управляет встречами | Гость бронирует → владелец входит по паролю → видит бронь в списке (имя, email) → отменяет | Уведомление об отмене, встреча пропадает из списка, слот снова доступен гостям |
| 5 | Защита кабинета владельца | Открыть `/admin` без входа | Редирект на страницу входа |

## Что сознательно не покрыто e2e

- Валидация полей, границы периодов, пересечения правил доступности — это
  быстрее и стабильнее проверяется юнит-тестами бэкенда (`backend/test/`,
  10 сценариев через `fastify.inject`).
- Настройка доступности владельцем через UI — второстепенный сценарий,
  API-слой покрыт тестами бэкенда; добавим e2e при изменении этого экрана.

## Замечания по стабильности

- Тесты выполняются последовательно (`workers: 1`): бэкенд один на весь прогон,
  данные в памяти общие.
- Каждый тест использует уникальные имя/email гостя и «ближайший будний день»,
  чтобы не зависеть от порядка выполнения и текущей даты (расписание сида:
  Пн–Пт 10:00–18:00 Москвы).
